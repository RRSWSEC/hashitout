# Contributing to Hash-It-Out

If you're here to make this better - good. That's the point.  Have a specific solve this failed? Know that it has an answer but either never found it or know what it is, but hashitout didn't find it? We would LOVE this info.  Reach out directly to thementor@rapidriverskunk.works or aven@rapidriverskunk.works If you've got someting to share, or do it through github.

If you're here to fork it, rename it, and pretend you built it - you're in the wrong place.

This project is maintained by **RRSW Corp (Non-Profit)** and exists to support real-world CTF work, analysis, and learning without losing authorship or turning into fragmented clones.

## First - read this

Before doing anything:

- Read the LICENSE

---

## What we actually want

Contributions should make the tool:

- More accurate
- More useful in real scenarios
- Easier to understand
- Harder to misuse
- Honest about what it can and can't do

Good contributions:

- Fix real bugs
- Improve detection or decoding logic
- Handle edge cases cleanly
- Improve output clarity
- Add meaningful features (not noise)
- Add solid test cases
- Improve documentation without fluff

---

## What we do NOT want

Don't submit:

- Feature bloat
- "Looks smart" code that lies or guesses silently
- Obfuscated or over-clever code
- Rebrands disguised as contributions
- Breaking changes without a strong reason
- Anything that strips attribution or project identity

If it makes the tool look cooler but less trustworthy, it's getting rejected.

---

## Core principles

Keep it:

- Practical over flashy
- Clear over clever
- Honesty over ego
- Traceable over magical

If a decoder is uncertain, say it's uncertain. Don't fake confidence. Even better, fix it! Small, light, and neat - preferably.

---

## How to contribute

1. Open an issue if it's a big change
2. Fork the repo
3. Create a clean branch
4. Make your changes
5. Submit a PR with:

   - What you changed
   - Why it matters
   - What problem it solves
   - Any edge cases considered
   - Example input/output if applicable

If I have to reverse-engineer your intent from your code, it's getting bounced.

---

## Code expectations

- Keep it readable
- Keep it scoped
- Don't break existing behavior unless necessary
- Comment anything non-obvious
- No silent failures that aren't disclosed for the technical documentation and transparency

## Forks vs contributing

Forks will happen. That's fine.

But understand this:

> If you build something better and keep it to yourself, you helped yourself.
> If you contribute it back, you helped everyone.

This project is meant to grow as a shared tool, not splinter into 50 half-maintained versions.

If your change belongs here, contribute it here.
